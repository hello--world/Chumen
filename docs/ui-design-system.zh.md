# Chumen UI 设计系统

English version: [ui-design-system.en.md](ui-design-system.en.md)

本文档是 Chumen UI 实现级设计契约。[DESIGN.zh.md](../DESIGN.zh.md) 定义产品方向；本文档定义具体视觉和交互规则。PIN、配置加密、应用锁屏、Keychain、运行明文、日志和 AI 审核边界以 [security-model.zh.md](security-model.zh.md) 为准。

## 设计目标

Chumen 是用于操作 mihomo 的 macOS 原生工具。它应该更接近系统设置、快捷指令和专注的开发工具，而不是网页仪表盘。

默认视觉应该安静、系统原生。允许使用颜色，但只在颜色能改善识别或操作优先级时使用。布局要保持足够紧凑，适合重复使用。

## 必须遵守的规则

- 优先使用 SwiftUI/AppKit 系统控件。
- 使用 macOS 语义颜色表达表面和文本。
- 圆角矩形保持 8 pt，除非系统控件自带形状。
- 不要卡片套卡片；使用分组行、section、分割线或提示条。
- 不要用大面积装饰渐变作为页面背景。
- 不要把重要决策说明写成很小的灰字。
- 默认窗口中，首次启动流程的关键操作不能藏在必须滚动之后。
- 修改布局、颜色或首次启动流程后，必须截图检查可见 UI。

## 颜色系统

### 中性表面

按层级使用：

- 页面：`Color(nsColor: .windowBackgroundColor)`
- 主表面：`Color(nsColor: .textBackgroundColor)`
- 分组区域：`Color(nsColor: .controlBackgroundColor)`，需要时降低不透明度
- 边框：低不透明度 `Color(nsColor: .separatorColor)`
- 主文本：`.primary`
- 次级文本：`ChumenStyle.mutedText` 或 `Color(nsColor: .secondaryLabelColor)`

大面板、表单、列表和设置分组应保持中性。页面不应读起来像单一蓝色、紫色、米色、橙色、棕色或深蓝灰主题。

### 强调色

- 蓝色：主操作、焦点搜索、选中 segmented control。
- 绿色：运行、健康、启用状态。
- 红/橙：破坏性、警告或风险取舍。
- 青绿、紫、橙：小图标块、分类提示或局部视觉锚点。

快捷指令式颜色可以用于小身份块：图标、badge、紧凑提示。不能用于大背景、页面背景或只是展示信息的全宽卡片。

### 渐变

渐变只允许用于：

- 应用/安全图标块；
- 很小的分类 badge；
- 不和控件竞争的空状态插图。

不要把渐变放在表单、表格、设置分组或常规仪表盘指标后面。

## 字体

优先使用系统 text styles，不随意硬编码字号，除非 UI 是固定格式状态值。

- 页面/弹窗标题：`.title3.weight(.semibold)` 或 `.title2.weight(.semibold)`
- section 标题：`.headline` 或 `.subheadline.weight(.semibold)`
- 主表单/控件文本：`.body` 或 `.callout`
- 次级解释：重要时用 `.callout`，低优先级提示才用 `.caption`
- 指标值：`.title2.weight(.semibold)` 或 `.title3.weight(.semibold)`

重要安全或配置说明不能使用 caption 尺寸。用户必须理解后果时，至少使用 `.callout`。

## 布局节奏

- 外层页面 padding：20 到 28 pt。
- 分组 padding：12 到 16 pt。
- 表单行间距：8 到 12 pt。
- section 间距：12 到 18 pt。
- 默认 sheet 最大宽度：680 到 840 pt，按内容决定。
- 默认圆角：`ChumenStyle.radius`，8 pt。
- 常规阴影应没有或极弱。

分组内部用分割线表达层级。避免多个带边框盒子再堆进另一个带边框盒子。

## 组件规则

### 设置表单

设置页应像原生分组表单：

- 简单设置左侧 label、右侧 control；
- 每个概念一个分组；
- 需要时在控件下方放解释；
- 普通编辑自动保存，除非对运行中内核生效必须是独立动作。

避免在大空白区域放孤立按钮。动作要放在相关设置附近或明确 footer 行里。

### 首次启动安全设置

这个流程保护 age 私钥，并在首次导入配置前运行。产品语义必须遵守 [security-model.zh.md](security-model.zh.md)，UI 不得把 PIN 保护解释成应用锁屏。

默认行为：

- 配置文件默认加密；
- age 私钥默认启用 PIN 保护；
- 应用锁屏是独立选项，默认关闭；
- 用户可以不用 PIN 继续，但文案必须说明配置仍会加密。

布局规则：

- 默认窗口内必须能完成，不应要求滚动；
- 左侧说明目的，一个图标和短文案即可；
- 右侧放实际控件；
- 私钥保存位置是紧凑行和 segmented picker；
- 默认只显示一个生成 PIN 字段；
- 生成 PIN 默认可见，因为是应用创建给用户看的；
- 提供眼睛切换和重新生成图标按钮；
- 不要显示第二个一模一样的确认字段，除非用户明确进入自定义密码流程；
- 应用锁屏是 PIN 字段下方的独立 toggle；
- 主操作是 `启用 PIN 加密`；
- 次操作是 `不用 PIN，继续`；
- 说明是提示，不是操作：放左侧摘要或轻提示条，不要做成像可点击卡片的样式。

用户应能立刻回答：

- 保护的是什么？
- age 私钥保存在哪里？
- 生成的 PIN 是什么？
- 应用锁屏是否开启？
- 不用 PIN 继续会怎样？

### 启动导入

没有配置时，安全设置之后显示导入提示。

- 标题直接说明：还没有代理配置。
- 主操作：扫描/导入已有配置。
- 次操作：导入本地 YAML 或稍后继续。
- 扫描结果列表必须提供搜索，支持按配置名称、来源客户端和路径过滤。
- 搜索状态下的“导入全部”只作用于当前过滤结果，避免用户误导入隐藏项。
- 导入配置的显示名称不要保留原应用名；来源应用应放到 metadata 或备注。

### 配置页

配置页是配置库管理界面，不是把所有导入表单和所有配置动作平铺到同一层的页面。

- 左侧是入口栏：新建配置、导入本地 YAML、导入订阅、扫描其他客户端、全局覆写。
- 新建配置必须是第一优先级入口；点击后创建最小可运行 YAML，并直接打开编辑器。
- 右侧是配置列表；每个配置是一行可扫描信息，不做大卡片。
- 行内只放高频动作：编辑、编辑规则、编辑节点、更新、更多。
- 低频或危险动作放进“更多”：编辑代理组、扩展覆写、扩展脚本、打开文件、代理更新、删除。
- 当前配置只显示一个“当前启用”状态；不要同时显示“已启用”和“当前启用”。
- 来源客户端、订阅 URL、文件路径是 metadata，不参与主标题。
- 配置列表需要本页搜索，按名称、路径、来源客户端和订阅 URL 过滤。

### 搜索

头部搜索是入口。激活后打开 Spotlight 风格覆盖层。

- 激活层必须盖住入口，不能同时看到两个搜索框。
- 结果优先级：设置/内核，其次配置、代理、Provider、规则、连接、日志。
- 搜索必须 debounce，不能阻塞输入或中文 IME 组合输入。
- 空结果数字必须等于实际显示结果，而不是隐藏总数。

### 仪表盘和指标

- 指标是可扫描 tile，不是营销卡片。
- 流量和速率值需要稳定尺寸。
- 不使用大面积单色块。
- 状态色按状态：健康用绿，关闭用灰，失败用橙/红。

### Provider 和代理列表

- 列对齐必须一致。
- 策略选择器列宽一致。
- 操作按钮保留文字标签，除非功能由熟悉符号即可明确。
- 长列表使用低对比行背景和分割线，不要每行都做悬浮卡片。
- 下拉菜单不能模糊、裁切或露出后方内容。

### AI 助手

- 没有可用模型 endpoint/key 时，AI 助手作为本地搜索。
- 模型生成操作只能产生待审核 diff。
- 用户应用前，不得直接修改配置、规则、代理/TUN、系统代理或运行配置。

## 可访问性和验证

每次 UI 修改完成前检查：

- 默认窗口大小下文字不溢出；
- 首次启动流程的关键操作不藏在滚动之后；
- 焦点 ring 不出现在覆盖层后面；
- 按钮有标签或 tooltip；
- 重要说明在截图尺度下可读；
- 颜色不是唯一状态信号；
- 逻辑变更后 `swift test` 通过；
- 可见 UI 变更已截图检查。

## 维护

当 UI 决策和本文冲突时，先更新本文，或在组件附近写清楚例外原因。后续 agent 应把本文视为详细 UI 实现的设计来源。
