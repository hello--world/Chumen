## 品牌

- 性格：安静、原生、技术向、可靠。
- 信任信号：状态清晰、控制可逆、核心/API 反馈直接、中文文案可读。
- 避免：网页仪表盘式卡片堆叠、厚重灰色横幅、强阴影、装饰性渐变、照搬 macOS 设置布局。

## 产品目标

- 目标：让内核状态、代理状态、TUN、流量和常用操作容易扫描和控制。
- 非目标：主题定制和营销式装饰 UI。
- 成功信号：应用感觉像一个小型 macOS 工具，而不是嵌在窗口里的网页。

## 用户和任务

- 主要用户：在 macOS 本机运行 mihomo 的用户。
- 用户任务：启动/停止内核、查看状态、切换策略、管理 Provider、诊断连接、调整运行设置。
- 使用场景：菜单栏快速检查、反复查看仪表盘、偶尔进行高级配置。

## 信息架构

- 主导航：顶部标签栏。
- 核心页面：总览、配置、代理、Provider、连接、规则、内核、内核工具、日志、设置。
- 内容层级：当前运行状态优先，其次全局搜索、操作、关键指标、诊断摘要和可跳转的相关信息。
- 顶部 header：保持一行外壳，左侧应用和运行状态，中间 modest-width 搜索入口；状态信息按语义上下分组，`API` 在 `配置更新` 上方，`系统代理` 在 `模式 + TUN` 上方，避免横向排成过长状态带。
- 宽屏节制：总览页主内容和 header 主信息使用统一内容宽度，默认按可用宽度约 80% 展示，垂直节奏增加约 20%，不铺满超宽窗口。
- 搜索：header 搜索只是入口；激活后打开 Spotlight 风格覆盖层，必须盖住原搜索入口和 header 状态，不能出现两个搜索框。结果优先展示设置和内核，再展示配置、代理、Provider、规则、连接、日志。
- AI：右下浮动入口打开小聊天窗口；本地 Ollama 是优先路径，默认地址 `http://127.0.0.1:11434/v1`，不需要 Key。远程 OpenAI-compatible endpoint 需要保存 Key。没有可用模型时只作为本地搜索。
- 设置边界：`内核` 管 mihomo/runtime 配置，`设置` 管 Chumen 自身偏好。普通设置自动保存；对运行中内核生效的保存项需要明确应用动作。

## 设计原则

- 使用低对比分组面和细边框，不使用突出的悬浮卡片。
- 状态可读，但不靠巨大色块或强颜色表达。
- 优先使用 SwiftUI/AppKit 原生控件和语义颜色；自定义颜色只作为少量身份或操作强调。
- 功能密度比装饰留白更重要。
- 仪表盘必须是模块化指挥台：各功能模块通过 Dashboard provider 暴露关键状态、诊断和跳转入口，页面只负责排序和渲染，不直接堆叠业务定制卡片。

## 视觉语言

- 颜色：系统自适应背景；绿色只表达运行/健康；蓝色只表达主操作或焦点。
- 字体：系统字体；状态值用 semibold；标签用 caption/callout。
- 间距：紧凑工具型节奏；分组和列表保持一致。
- 圆角/阴影：常规圆角最大 8 pt；普通仪表盘表面不使用重阴影。
- 动效：系统默认即可。
- 图标：SF Symbols 和应用图标。菜单栏状态图标使用门的状态隐喻：关闭代表未捕获流量，半开代表系统代理，打开代表 TUN 有效路由。

## 组件

- 复用组件：header、命令面板、Dashboard item/section、设置表单、顶部标签导航。
- 新/重点组件：全局搜索覆盖层、紧凑 header 状态条、模块化仪表盘 provider、内核快速导航、独立设置表单、首次启动安全设置、AI 待审核面板、菜单栏状态图标。
- 状态：运行/停止、API 可达/不可达、代理开/关、TUN 开/关、空指标、本地 Ollama 可用、远程 AI 缺 Key、AI 待审核 diff。
- 样式 token：`ContentView.swift` 里的 `ChumenStyle`。

## 可访问性

- 目标：默认 macOS 字号下清晰可读。
- 键盘/焦点：使用系统控件；搜索支持 Return 打开首个结果、Esc 或外部点击关闭；搜索结果 debounce，不阻塞中文 IME，不在主线程扫描大列表。
- 对比度：避免在 material blur 上放低对比文字。
- 辅助语义：按钮和状态值必须有标签。
- 动效：避免非必要动画。

## 响应式

- 支持可调整大小的 macOS 窗口。
- 仪表盘 section 和 item 用自适应网格换行；同一模块可以提供状态、指标、诊断或链接，但必须给出稳定标题、值、优先级和动作语义。
- header 外壳保持一行；状态 pill 内部可以上下分组。窄窗口优先收紧搜索和状态组宽度，不隐藏 `API`、配置更新、系统代理、模式和 TUN。
- 宽屏窗口下，总览主内容不应横向铺满；使用 `ChumenStyle.shellContentWidth(for:)` 控制扫描宽度，默认宽度约为可用宽度的 80%。

## 交互状态

- 加载：保留已有状态和最后刷新值。
- 空状态：使用短横线占位。
- 错误：常见连接失败必须本地化，技术细节放日志或工具里。
- 成功：状态文字简短。
- 禁用：使用原生禁用样式。
- AI 待审核：所有 AI 操作都是临时提案，必须有类似 git 的 diff。用户显式应用前，不得直接修改配置、规则、代理/TUN、系统代理或运行配置。

## 安全和隐私模型

- 权威文档：`docs/security-model.zh.md`。
- 配置默认用 mihomo age 路径加密；不要再引入另一套长期配置加密格式。
- PIN 有两个独立含义：age 私钥保护和可选应用锁屏。PIN 保护默认开启；应用锁屏默认关闭。
- 默认启动应通过本地包装密钥自动解锁 PIN 保护的 age 私钥，让软件像普通应用一样打开。
- 开启应用锁屏必须删除 auto-unlock 路径并要求启动输入 PIN。关闭应用锁屏只能在 age 私钥已解锁时恢复 auto-unlock。
- 不要把应用锁屏解释或实现为启用 PIN 的副作用。
- 默认 age 私钥存储是本地文件；Keychain 是可选项。
- 不要用固定 salt 做默认解密机制。PIN 派生使用随机 per-vault salt；默认 auto-unlock 使用随机本地包装密钥。
- 固定 Application Support runtime config 在配置保护开启时只能保存受保护数据。 unavoidable 明文运行材料必须写到随机临时会话目录并由专门清理逻辑清除。
- 所有用户可见的安全/运行通知必须同步写日志；禁止记录 PIN、API Key、controller secret、age 私钥、解密配置和带凭据订阅 URL。

## 文案

- 语气：简洁、工具型。
- 术语：统一使用 core、API、TUN、DNS、Provider。
- 中文 UI 不应泄漏常见 Foundation 网络错误英文。

## 实现约束

- 框架：SwiftUI/AppKit 原生控件。
- token：样式 token 放在 `ChumenStyle`，不要新增主题层。
- UI 变更前先检查 `docs/ui-design-system.zh.md`；需要例外时，先更新文档或在组件附近写代码意图注释。
- Dashboard 新增内容必须优先实现为 provider/item 数据，不要直接在 `DashboardView` 内硬编码模块专属视图；日志、连接、规则、Provider 等模块可以各自注册重要摘要和跳转入口。
- Dashboard 快捷按钮同样必须由 provider/item 发布；启动、停止、刷新、系统代理、TUN、设置入口和模块跳转都属于同一套快捷入口协议。
- Dashboard 快捷控制必须按优先级分层：启动、停止、重启、刷新、系统代理、TUN 和编辑快捷控制固定在第一行；用户扩展出来的启动/退出偏好、网络选项、模块跳转等只能排在下方扩展行，不能挤乱核心操作。
- 性能：流量和连接流更新时，仪表盘必须保持轻量。
- 安全：AI API Key 存 Keychain；本地 Ollama 不需要 Key；模型调用使用用户配置的 OpenAI-compatible endpoint。
- 验证：样式改动后运行 Swift 构建/测试；可见 UI 变更尽量截图检查。
